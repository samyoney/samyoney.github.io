When I first started using AI at work, I put almost every instruction into the prompt. That worked for small tasks. As soon as a task crossed repositories, project data, source code, and customer-facing communication, the prompt became long and easy to forget.

The bigger issue was that the prompt only lived in one conversation. On the next task I had to explain the same things again: which source was authoritative, what needed approval, how Japanese content should be reviewed, and which test counted as evidence.

That is why I started using `AGENTS.md`.

## A small set of working rules

I do not treat `AGENTS.md` as a giant prompt. It feels closer to a repository handbook. The root file contains a few shared rules, such as keeping changes within scope, checking the latest source before an update, and verifying the result before reporting completion.

When one folder needs a different workflow, I place another `AGENTS.md` closer to it. A project-management folder may require a fresh data fetch before editing. A source folder may require checking references and running a focused test. The closest relevant rule takes priority.

This makes the guidance specific without repeating everything in every directory.

## What I keep in the file

I only keep rules that are stable and testable:

- what belongs in the folder;
- which system is the source of truth;
- when confirmation is required;
- which tool or workflow should be used;
- what evidence is needed before a task is complete.

Temporary task history, secrets, and machine-specific paths belong elsewhere. If the file tries to describe everything, it quickly becomes another document nobody trusts.

I also avoid vague rules such as “be careful.” A useful rule should change the next action. “Read the latest WBS before updating a task” is much easier to follow and verify.

## What changed for me

After adopting this approach, I spent less time repeating the same operational context. More importantly, AI stopped behaving like a tool dropped into an unfamiliar repository and started working more like a new team member who had read the local handbook.

`AGENTS.md` does not make AI smarter. It gives that capability a clear place to operate. For real project work, that difference is often more valuable than adding another long paragraph to a prompt.

そこで使い始めたのが `AGENTS.md` です。

## リポジトリの小さな仕事ルール

私は `AGENTS.md` を巨大なプロンプトにはしていません。リポジトリの簡単な作業ガイドに近いものとして扱っています。ルートには、変更範囲を広げないこと、更新前に最新データを確認すること、完了を伝える前に結果を検証することなど、共通のルールだけを書きます。

特定のフォルダに別の進め方が必要なら、その近くにもう一つ `AGENTS.md` を置きます。管理データのフォルダでは編集前の再取得を求め、ソースコードのフォルダでは参照先の確認や関連テストを求める、といった使い分けです。作業対象に最も近いルールを優先します。

この形なら、同じ説明を何度も繰り返さずに、必要な場所だけ具体的にできます。

## 書いていること、書かないこと

残すのは、長く使えて確認できるルールです。

- そのフォルダに置くもの
- 正とするデータやシステム
- 事前確認が必要な場面
- 選ぶべきツールや進め方
- 完了前に必要な確認結果

一時的なタスク履歴、秘密情報、特定PCだけのパスは別の場所に置きます。何でも書こうとすると、すぐに古くなり、読まれない文書になってしまうからです。

「注意して作業する」のような曖昧な表現も避けています。「タスクを更新する前に最新のWBSを取得する」のように、次の行動が決まる書き方のほうが実際に役立ちます。

## 使ってみて変わったこと

この形にしてから、同じ前提を何度も説明する時間が減りました。それ以上に、AIが何も知らずにリポジトリへ入ってくるツールではなく、最初に作業ガイドを読んだ新しいメンバーに近い動きをするようになりました。

`AGENTS.md` がAIそのものを賢くするわけではありません。AIがどこで、どのように力を使うかを明確にするものです。実際のプロジェクトでは、長いプロンプトを追加するより、この違いのほうが大きいと感じています。
