## Introduction

This sample shows how I structure an iOS application with SwiftUI, MVVM, and Clean Architecture principles. The application is intentionally compact so that its state flow, dependency boundaries, and testing strategy remain easy to follow.

## Why SwiftUI?

[SwiftUI](https://developer.apple.com/swiftui/) is Apple's declarative framework for building interfaces across its platforms. Views describe the desired UI for the current state, while the framework manages the updates required when that state changes.

For this sample, SwiftUI offers several practical benefits:

- **Readable state-driven views:** screen behavior is easier to trace from state to output.
- **Reusable components:** small views can be composed into larger features.
- **Efficient iteration:** previews provide a short feedback loop during UI development.
- **Platform integration:** SwiftUI works with the broader Apple SDK and can coexist with UIKit when needed.

Sharing SwiftUI concepts across Apple platforms is useful, but production interfaces still need platform-specific navigation, interaction, and layout decisions.

## Architecture approach

Clean Architecture is used here as a dependency discipline rather than a rigid template:

1. **Presentation:** SwiftUI views render state, while ViewModels coordinate user actions and screen state.
2. **Domain:** entities and use cases express business behavior without depending on UI or storage frameworks.
3. **Data:** repository implementations coordinate APIs, persistence, and mapping.
4. **Infrastructure:** concrete Apple frameworks and third-party libraries remain behind interfaces where substitution or testing matters.

MVVM does not define use cases by itself. In this sample, MVVM handles presentation concerns, while use cases provide an optional domain boundary for operations that deserve independent testing or reuse.

## Technology choices

The sample uses SwiftUI for the interface and demonstrates integrations such as [SwiftData](https://developer.apple.com/documentation/swiftdata), [R.swift](https://github.com/mac-cain13/R.swift), and [Resolver](https://github.com/hmlongco/Resolver). These are implementation choices for the reference project, not requirements for Clean Architecture.

## Repository structure

The project is organized into five main areas:

- **App:** application lifecycle, dependency composition, and navigation.
- **Core:** shared utilities and infrastructure such as networking, preferences, caching, and reusable views.
- **Presentation:** SwiftUI screens and their ViewModels.
- **Resources:** localized strings, assets, and animations.
- **Data:** repositories, API clients, persistence, mapping, and domain operations used by the sample.

The important boundary is the dependency direction: presentation code should not know how an API request or database query is implemented. That separation makes ViewModel and use-case tests deterministic and keeps framework changes localized.

## Testing and trade-offs

Unit tests should cover business operations, repository behavior, and ViewModel state transitions. UI tests are most effective when reserved for critical journeys such as authentication and primary navigation.

For a small feature, separate use-case and repository layers may add more ceremony than value. The structure should grow with the product; boundaries are useful when they isolate change, ownership, or risk.

## Conclusion

This starter provides a practical baseline for SwiftUI applications: explicit state, clear dependency boundaries, and testable behavior without tying the domain to a specific UI or persistence framework.
