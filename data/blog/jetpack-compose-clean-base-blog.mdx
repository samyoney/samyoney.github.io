## Introduction

This sample shows how I structure an Android application with Jetpack Compose, MVVM, and Clean Architecture principles. It is intentionally small: the goal is to make the dependency flow, state management, and testing boundaries easy to understand before adding production complexity.

## Why Jetpack Compose?

[Jetpack Compose](https://developer.android.com/develop/ui/compose) is Android's modern declarative UI toolkit. UI is described as composable functions that react to state, which reduces the amount of synchronization code normally required between views and application state.

For this sample, Compose provides three practical advantages:

- **State-driven UI:** each screen renders from an explicit UI state.
- **Reusable components:** common elements can be composed without inheriting from framework classes.
- **Faster iteration:** previews and concise Kotlin code shorten the feedback loop.

## Architecture approach

Clean Architecture is an organizing approach, not a mandatory framework. The useful part is the dependency rule: business decisions should not depend directly on UI, storage, or network implementations.

The sample follows the same broad separation described in the [Android architecture recommendations](https://developer.android.com/topic/architecture/recommendations):

1. **Presentation:** Compose screens and ViewModels expose observable UI state and handle user actions.
2. **Domain:** use cases express application-specific operations when that extra layer adds clarity.
3. **Data:** repositories coordinate local and remote data sources behind stable interfaces.
4. **Framework:** Android-specific storage, networking, and platform integrations remain at the boundary.

This separation improves testability and makes implementation details easier to replace. For a very small application, some layers can be combined; architecture should reduce complexity rather than create ceremony.

## Repository structure

The repository is organized around three top-level areas:

- **App:** application entry points, navigation, Compose UI, ViewModels, use cases, and data contracts.
- **buildSrc:** centralized build configuration and dependency versions used by this sample.
- **Framework:** reusable Android-specific implementations for persistence, preferences, caching, and shared base types.

Within the app, presentation code depends on domain contracts rather than concrete network or database classes. Repositories hide data-source details, and ViewModels translate results into immutable UI state. This keeps screen behavior testable without rendering the UI or calling real services.

## Testing and maintenance

The most valuable tests target use cases, repository behavior, and ViewModel state transitions. Compose UI tests can then focus on critical user journeys. This balance gives fast feedback while avoiding fragile tests for every visual detail.

When extending the starter, I recommend adding abstractions only when there is a real boundary to protect. A clear two-layer feature is better than a five-layer design with pass-through classes.

## Conclusion

This starter demonstrates a maintainable baseline for Compose projects: state-driven UI, explicit dependencies, testable business operations, and framework code kept at the edges. It is a reference implementation, so the structure should be adapted to the size and risk profile of the product.
